### 3.2 高频问题分类

**🔴 问题 1: Agent 自行停止（最高频）**
- 在 hermes-chat 迭代 session 中出现 5+ 次
- 用户必须反复说"怎么停了"、"为什么停了"
- 根因: max_turns 到达上限，或 context compaction 后 agent 失去动力
- 修复: 自主模式下应主动报告"即将达到 turn 上限，需要继续"

**🔴 问题 2: 方向偏离**
- 做成"通用 OpenAI 兼容 chatbot"而不是"Hermes 原生客户端"
- 根因: 缺少持续的意图对齐检查
- 修复: 每 5-10 个 turn 做一次 self-check

**🟡 问题 3: Git commit/push 需要提醒**
- 用户得说"记得 git push"
- 根因: 没有把 commit/push 内化为习惯
- 修复: 每完成一个阶段性目标就自动 commit+push

**🟡 问题 4: Tool 错误率高**
- 最大 session 中 348/1029 = 34% 的 tool call 有错误输出
- 常见: patch 找不到目标字符串、terminal 命令语法错误
- 修复: patch 前先 read_file 确认内容，terminal 命令先 dry-run

## 四、工具使用模式

### 4.1 最常见工具组合
```
patch → patch → patch (×66)     # 连续代码修改
terminal → terminal → terminal (×46)  # 连续命令执行
patch → patch → terminal (×44)  # 改代码 + 验证
patch → terminal → terminal (×36) # 改一行 + 跑两个命令
```

**发现**: Agent 偏好"改一堆代码再验证"而不是"改一点验一点"。
这导致错误累积，一个 patch 出错后后面的全错。

## 五、经验提炼

### ✅ 做得好的

1. **代码修改效率高**: patch 是最常用工具，说明 agent 能精确定位并修改代码
2. **验证意识**: 每次修改后都会跑 tsc/bundle 验证
3. **浏览器验证**: 用 browser_vision 看实际效果，不只是看代码
4. **响应用户反馈**: 当用户说"太复古了"能快速调整方向
5. **Git 操作**: 能正确执行 git add/commit/push

### ❌ 做得不好的

1. **自主停止**: 到达 max_turns 就停，不主动告知用户
2. **方向漂移**: 做着做着就偏离核心需求
3. **批量修改风险**: 一次改太多文件，出错后难以回滚
4. **Tool 错误不感知**: 34% 的 tool call 有错误但继续执行
5. **缺少阶段性汇报**: 长时间不汇报进度，用户焦虑

### 🔧 可复用的模式

1. **修改-验证循环**: patch → tsc --noEmit → browser_vision
2. **意图对齐**: 每 5-10 turn 自问"我还在做用户要的东西吗？"
3. **渐进式提交**: 每完成一个功能点就 commit+push
4. **错误感知**: tool 返回 error 时立即处理，不累积
